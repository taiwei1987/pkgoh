# pkgoh 产品需求文档（PRD）

版本：v1
日期：2026-04-28
项目：pkgoh
定位：macOS 终端资产管理器（TUI）

## 1. 产品概述

`pkgoh` 是一个面向 macOS 开发者的终端资产管理器。

用户在日常开发中，往往会通过多个包管理器安装大量全局工具、运行时、CLI 和桌面应用，例如：

- Homebrew
- npm
- pnpm
- cargo
- pip
- uv
- mas

随着使用时间增长，用户会逐渐失去对这些工具的整体认知：

- 不知道自己到底装了什么
- 不知道哪些工具最占空间
- 不知道哪些工具长期没用
- 不知道哪些工具删掉风险较低
- 不知道清理后大概能释放多少空间

`pkgoh` 的目标，是通过一个键盘优先、信息密度高、反馈明确的 TUI 界面，让用户在终端里完成统一扫描、查看、筛选、评估、删除和清理缓存。

## 2. 产品目标

### 2.1 核心目标

让用户安装 `pkgoh` 后，只需输入：

- `pkgoh`
- 或简写 `pkg`

即可进入一个统一的终端界面，快速完成：

- 扫描终端生态中安装的工具
- 按体积、使用频率和来源理解资产分布
- 快速定位特定工具
- 批量选择目标项
- 删除不需要的工具
- 清理缓存
- 在执行前理解风险与收益

### 2.2 产品原则

- 必须是真实扫描，不是静态演示
- 必须是真实操作，不是假删除
- 必须有持续反馈，不能让用户觉得卡死
- 必须尽量降低误操作风险
- 必须优先适配非技术用户的理解方式
- 必须保持终端环境下的高效率与低心智负担

## 3. 目标用户

### 3.1 主要用户

- 在 macOS 上进行开发的个人用户
- 经常安装 CLI、运行时、开发依赖的用户
- 对终端有使用习惯，但不一定精通系统维护

### 3.2 典型用户特征

- 会通过命令行安装工具
- 机器上可能同时存在多个包管理器
- 对“哪些东西能删、删了会不会出问题”缺乏确定性
- 希望界面直观，而不是依赖大量命令记忆

## 4. 核心使用场景

### 场景 1：盘点终端资产

用户想知道自己机器上到底安装了哪些全局工具、运行时和应用。

### 场景 2：查找大体积工具

用户磁盘空间紧张，想快速找出最占空间的项目。

### 场景 3：找出长期不用的工具

用户想清理很久没碰过的工具，但又担心误删关键依赖。

### 场景 4：安全删除

用户想删除工具，但希望系统提前给出“风险等级”和简明说明。

### 场景 5：清理缓存

用户不一定想卸载工具，只想先清缓存回收空间。

### 场景 6：快速搜索和定位

用户记得工具名字，但不知道它属于哪个来源，希望输入关键字快速定位。

## 5. 当前支持的资产来源

当前版本必须支持扫描以下来源：

- Homebrew formula
- Homebrew cask
- npm 全局包
- pnpm 全局包
- cargo 安装的二进制
- pip 全局包
- uv 管理的 Python 运行时
- uv tool 安装的工具
- mas 管理的 Mac App Store 应用

说明：

- 扫描不应仅局限于“单一标准目录”
- 对于通过 PATH 暴露、但实际安装在非标准全局目录中的 Node 工具，也应尽可能纳入识别
- 扫描目标是“支持生态内尽可能完整”，而不是盲目全盘扫描整个硬盘

## 6. 必须展示的信息

每条资产至少要展示：

- 序号
- 名称
- 来源
- 版本
- 空间大小
- 最后使用时间

右侧详情面板至少要展示：

- 名称
- 来源
- 版本
- 大小
- 最近使用时间
- 标签（如大体积、长期未用）
- 评估建议
- 评估说明
- 是否可清缓存
- 工具简介 / 功能说明
- 路径信息 / 安装位置 / 缓存位置等

## 7. 风险评估模型

删除建议必须分为三层：

### 7.1 可删除

定义：

- 没有明显依赖或被依赖关系
- 删除后通常只影响当前工具本身
- 恢复成本较低

### 7.2 建议保留

定义：

- 没有强依赖，但可能是工作流中常用基础工具
- 或存在一定引用关系，但影响相对可控
- 删除后可能会出现一些可修复的报错或命令失效

### 7.3 核心依赖

定义：

- 存在强依赖或明显被依赖关系
- 删除后可能影响其他工具、运行时或工作流
- 对非技术用户恢复成本较高

要求：

- 详情面板必须展示“评估说明”
- 文案必须尽量让非技术用户也能理解
- 评估是建议，不应伪装成 100% 保证

## 8. 关键交互要求

### 8.1 启动与加载

启动后必须立即进入 Loading 状态。

要求：

- 不能空白卡住
- 不能像程序无响应
- 必须有可见的加载动画（Spinner 或进度条）
- 必须显示当前扫描状态
- 中文系统显示中文文案，英文系统显示英文文案

当前简化要求：

- 核心文案可以是“加载中”或对应英文
- 不必展示过度技术化的扫描细节

### 8.2 列表导航

- 支持上下键移动
- 支持数字跳转
- 支持空格多选
- 当前选中项必须有明确焦点视觉反馈

### 8.3 搜索

- 支持按 `/` 进入搜索模式
- 输入关键字后实时过滤
- 支持按名称快速定位工具
- Esc 可退出或清空搜索

### 8.4 排序

- 默认支持按大小排序
- 用户可以通过快捷键再次触发排序
- 排序状态必须在顶部区域可见

### 8.5 多选与统计

当用户选中多个项目时，界面必须实时显示：

- 已选数量
- 预计可释放空间
- 当前资产总数

### 8.6 删除操作

要求：

- 必须先选中后删除
- 删除操作必须有确认动作
- 快捷键使用 macOS 用户易理解的 Delete 键
- 删除执行过程中必须持续给出反馈
- 删除成功后应优先更新当前列表，而不是每次都强制全量重扫

### 8.7 清缓存操作

要求：

- 必须先选中后执行
- 必须有确认动作
- 必须显示预计可释放空间
- 必须明确提示共享缓存可能按来源合并计算

### 8.8 刷新操作

- 支持快捷键 `R`
- 必须有二次确认
- 确认提示要足够明显，不能弱提示

### 8.9 退出操作

- 快捷键为 `Esc`
- 必须有确认动作
- 确认提示要足够明显
- 避免误触直接退出

## 9. 权限与密码体验要求

这是当前产品体验中的高优先级部分。

### 9.1 基本原则

- 不能让底层权限提示在 TUI 外层偷偷弹出
- 不能让用户看到“像卡死一样”的密码输入过程
- 密码输入、错误反馈、重试提示，应尽量在 TUI 内部统一呈现

### 9.2 如果操作需要管理员权限

必须做到：

- 明确提示当前操作需要管理员权限
- 明确告诉用户接下来应该做什么
- 如果密码错误，必须明确提示错误并允许重新输入
- 执行期间不能看起来像无响应

### 9.3 如果某些生态自身会触发底层权限流程

要求：

- 上层界面必须尽量接管流程
- 至少不能让用户处于“界面像卡死，但不知道系统在等什么”的状态

## 10. 本地化要求

界面应检测系统语言。

### 10.1 简体中文系统

- 优先显示简体中文
- 除必要英文（如包管理器名称、命令、版本号）外，说明类文案尽量中文化

### 10.2 英文系统

- 正常显示英文界面

### 10.3 本地化覆盖范围

至少包括：

- 顶部状态栏
- 底部快捷键说明
- Loading 状态
- 搜索提示
- 删除/清缓存/刷新/退出确认
- 权限输入与报错提示
- 详情面板中的说明类文案

## 11. 视觉与布局目标

这是本次希望重点重做的内容。

### 11.1 当前设计存在的问题

从产品角度看，当前 TUI 的问题包括但不限于：

- 信息层级还不够清晰
- 视觉节奏不够舒服
- 操作确认区域不够强
- 某些状态提示不够醒目
- 整体质感还不够“完成品”
- 有功能，但产品感和设计感不够强

### 11.2 新设计目标

希望新的 TUI 方案做到：

- 第一眼就知道这是“资产管理器”，而不是普通列表页
- 左侧列表与右侧详情信息层级更清楚
- 顶部摘要区更有结构感
- 操作区更像“控制台”，而不是普通文本框
- 危险动作、警告动作、普通动作的视觉区分更明显
- 搜索、选择、确认、执行中、完成后的状态转换更顺滑
- 更符合 macOS 开发者审美，但仍然是终端风格

## 12. TUI 约束（给设计时必须遵守）

这一部分非常重要。

新方案必须建立在“终端 TUI”现实限制之上，而不是 Web / GUI 假设。

### 12.1 基本约束

- 运行环境是终端，不是浏览器
- 以键盘操作为主，不依赖鼠标
- 不应依赖 hover
- 不应依赖复杂浮层体系
- 不应依赖多窗口或真弹窗
- 不应要求高分辨率图形能力
- 必须适应常见终端宽度变化
- 必须兼容中英文内容长度差异

### 12.2 状态约束

必须考虑以下状态的布局与体验：

- 首次加载
- 加载完成
- 空状态
- 搜索中
- 已多选
- 删除确认
- 清缓存确认
- 刷新确认
- 退出确认
- 权限输入
- 操作执行中
- 执行成功
- 执行失败

### 12.3 反馈约束

- 任何可能超过 300ms 的操作，都应有反馈
- 用户不能猜“现在程序是不是卡住了”
- 危险动作必须明显
- 错误提示必须明确且可恢复

## 13. 功能边界

### 13.1 本期必须实现

- 扫描支持来源
- 列表展示
- 搜索
- 多选
- 删除
- 清缓存
- 排序
- 风险评估
- 中文/英文本地化
- 权限流程优化

### 13.2 本期不强求

- 全盘文件系统扫描
- 任意来源的自动识别
- 鼠标优先交互
- 图形化图表
- 多页面复杂导航
- Web 风格组件堆叠

## 14. 成功标准

如果新的 TUI 设计是成功的，应该满足：

- 用户第一次打开就知道怎么用
- 用户不需要读很多说明也能完成删除或清缓存
- 用户能快速理解“哪些东西大、哪些东西久未使用、哪些东西风险高”
- 用户不会对权限流程感到困惑
- 用户不会轻易误删或误退出
- 用户主观感受是“这已经像一个成熟产品”，而不是 demo

## 15. 给设计工具 / 设计师的重点任务

请重点重做以下内容：

- 顶部摘要信息架构
- 列表区的视觉层级
- 详情区的组织方式
- 操作确认区的表现形式
- 搜索与过滤状态
- 危险动作与警告动作的视觉反馈
- 加载态与执行中态
- 中英文双语场景下的布局稳定性

## 16. 最终交付期望

设计方案输出时，最好至少包含：

- 主界面布局
- 加载态
- 搜索态
- 删除确认态
- 清缓存确认态
- 权限输入态
- 执行中态
- 空状态
- 错误状态

如果可以，最好补充：

- 区块职责说明
- 状态切换说明
- 快捷键提示布局建议
- 在窄终端宽度下的退化策略
