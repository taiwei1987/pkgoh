# pkgoh TUI 可实施重设计规范

版本：v1
日期：2026-04-28
适用范围：仅重做 TUI 的视觉与交互体验层，不改变现有核心能力

---

## 1. 设计目标

本规范用于指导 `pkgoh` 现有 Rust TUI 的整体重设计。

本次重设计的目标不是扩展功能，也不是修改产品能力边界，而是：

- 提升整体视觉完成度
- 提升信息层级清晰度
- 提升操作确认的安全感
- 提升搜索、选择、确认、执行中的状态体验
- 提升中英文混排下的布局稳定性
- 在保持真实扫描、真实删除、真实清缓存的前提下，让界面更像一个成熟产品

---

## 2. 本次改造的严格边界

### 2.1 允许修改的内容

- 布局结构
- 边框、分区、留白、对齐方式
- 颜色体系
- 标题样式
- 快捷键展示方式
- 焦点态 / 选中态 / 危险态 / 警告态的视觉表达
- 搜索模式的视觉表现
- Loading 界面的视觉表现
- 确认区和执行中的表现方式
- 成功 / 失败 / 提示消息的展示方式

### 2.2 不允许修改的内容

本次不能改变以下核心能力和语义：

- 仍然通过 `pkgoh` 或 `pkg` 进入 TUI
- 仍然扫描当前已支持的生态来源
- 仍然展示资产列表与详情面板
- 仍然支持多选
- 仍然支持搜索过滤
- 仍然支持数字跳转
- 仍然支持删除
- 仍然支持清缓存
- 仍然支持刷新
- 仍然支持退出确认
- 仍然支持中英文切换
- 仍然支持权限输入和错误反馈
- 仍然使用风险分级：可删除 / 建议保留 / 核心依赖

### 2.3 明确禁止新增的交互语义

以下内容本次不要引入：

- 不新增“Enter 打开详情页”
- 不新增“强制回收空间”这种新动作
- 不新增鼠标依赖
- 不改成 Y/N 主导的确认方式，保留现有快捷键确认逻辑
- 不改成多页面 / 多窗口式导航
- 不改成文件系统全盘扫描产品

---

## 3. 必须保留的当前交互语义

为了确保“只改体验、不改能力”，新的 TUI 设计必须兼容以下交互语义：

- `↑ / ↓`：上下移动焦点
- `Space`：选中 / 取消选中
- `Delete`：准备删除当前已选项目
- `C`：准备清理缓存
- `R`：准备刷新
- `/`：进入搜索模式
- `S`：按大小排序
- `Esc`：退出搜索、取消当前待确认动作，或进入退出确认
- `Enter`：确认当前待执行动作
- `0-9`：数字跳转

说明：

- `Enter` 在现阶段是“确认动作”，不是“打开详情页”
- 详情信息依旧常驻在右侧面板中
- 删除、清缓存、刷新、退出都必须是“准备态 -> 确认态 -> 执行态”

## 4. 视觉风格方向

### 4.1 风格关键词

- Terminal Modern
- Calm Control Surface
- Monochrome Base + Cyan Accent
- Clear Hierarchy
- Dense but Breathable

### 4.2 总体感受目标

用户打开后，应感受到：

- 这是一个终端里的“资产控制台”
- 不是普通的文本表格
- 信息很多，但不乱
- 危险动作和普通动作有明显层级区分
- 整体质感偏冷静、克制、专业

### 4.3 推荐色彩语义

建议沿用当前已确认的功能语义色，同时吸收新稿的视觉方向：

- 背景：深色近黑绿
- 主色：Cyan，用于主标题、焦点边框、进度条、活跃块强调
- 普通正文：浅灰白
- 次要信息：灰色
- `可删除`：绿色
- `建议保留`：黄色
- `核心依赖`：红色
- 删除确认：红色强调
- 刷新确认：青色强调
- 清缓存确认：黄色或青黄色强调

说明：

- `可删除` 不改成 Cyan，保持绿色
- `建议保留` 保持黄色
- `核心依赖` 保持红色

## 5. 布局架构

整个 TUI 保持四段式结构，但重新优化层次：

### 5.1 顶部摘要区（Header）

高度建议：2 行

#### 第一行

左侧：

- `PKGOH` 品牌标识
- 当前主状态（如：已加载 / 搜索中 / 删除确认 / 执行中）

右侧：

- 资产总数
- 已选数量
- 预计可释放空间
- 当前排序方式

#### 第二行

- 当前来源范围（固定写 All 或当前支持模块摘要）
- 当前搜索词（如果存在）
- 可以是轻量状态说明，不必太长

要求：

- Header 必须更有“控制台仪表区”感觉
- 不要只是把所有文字平铺到一行
- 信息优先级要清晰

### 5.2 主体区（Main Split View）

左右分栏，建议比例：6:4 或 65:35

#### 左侧：资产列表

必须包含这些列：

- 序号
- 选择状态
- 名称
- 来源
- 版本
- 大小
- 最近使用

要求：

- 表头固定
- 列表区独立滚动
- 当前焦点行必须非常明确
- 选中态与焦点态必须区分开
- 搜索命中项应有额外的轻量强调

#### 右侧：详情面板

详情面板继续常驻，不新增独立详情页。

建议结构：

1. 标题区
   - 工具名
   - 来源
   - 风险标签

2. 核心摘要区
   - 简介说明
   - 风险评估
   - 评估说明

3. 元数据区
   - 版本
   - 大小
   - 最近使用
   - 是否可清缓存

4. 路径信息区
   - 安装位置
   - 缓存位置
   - 相关路径说明

要求：

- 右侧不再像“原始文本堆叠”
- 要有清楚的分段
- 风险建议必须在详情区里足够突出

### 5.3 操作区（Action Panel）

现有右下角操作区保留，但要重设计成更强的“动作控制面板”。

其职责不是普通说明文本，而是：

- 当前待执行动作说明
- 当前确认状态
- 当前执行进度
- 当前权限提示
- 当前成功 / 失败反馈

要求：

- 这是整个界面中“状态变化最明显”的区域
- 危险操作时边框、标题、文案都必须明显升级
- 不能只是平铺 2-3 行文本

### 5.4 底部快捷键区（Footer）

高度建议：1-2 行

要求：

- 快捷键说明要分组展示
- 不要又长又挤成一条难读的句子
- 建议按“导航 / 选择 / 操作 / 退出”分组
- 使用方括号包裹按键是可以的

例如分组思路：

- `[↑↓] 移动` `[0-9] 跳转` `[ / ] 搜索`
- `[Space] 选择` `[Delete] 删除` `[C] 清缓存`
- `[R] 刷新` `[S] 排序` `[Esc] 返回/退出` `[Enter] 确认`

## 6. 状态设计要求

新设计必须完整覆盖以下真实状态，而不是只做默认态。

### 6.1 Loading 状态

要求：

- 居中展示
- 有明显加载动画或进度条
- 核心文案保持简洁：`加载中` / `Loading`
- 可以显示当前扫描来源，但不要让界面变成技术日志窗口
- 不强调“正在扫描哪个路径”

### 6.2 默认浏览态

要求：

- Header 清晰
- 列表与详情稳定
- Footer 易读
- 用户一眼能理解焦点在哪、详情看哪里、操作在哪里

### 6.3 搜索态

要求：

- 搜索输入状态要明显
- 搜索词在 Header 或列表顶部有独立展示
- 列表实时过滤
- 命中项可高亮，但不要过度花哨

### 6.4 多选态

要求：

- 已选数量和预计可释放空间必须突出
- 操作区应主动提示可执行动作

### 6.5 删除确认态

要求：

- 不使用真正脱离布局的“GUI 弹窗思维”
- 可以用居中覆盖层，也可以用强化版操作面板
- 必须明显表现“危险确认”
- 必须明确告诉用户：再次按 Delete 或 Enter 确认，Esc 取消
- 如果选中项中包含高风险项，必须明确提示数量和风险

### 6.6 清缓存确认态

要求：

- 视觉危险等级低于删除
- 仍需明显确认
- 清楚说明预计释放空间

### 6.7 刷新确认态

要求：

- 明显但不危险
- 明确告诉用户会重新扫描列表

### 6.8 退出确认态

要求：

- 必须比普通提示更显著
- 但危险等级低于删除
- 避免误触直接退出

### 6.9 权限输入态

这是必须重点设计的状态。

要求：

- 输入密码的区域必须非常明确
- 当前操作名必须显示
- 必须清楚提示用户下一步该做什么
- 密码错误时，错误反馈必须醒目但不混乱
- 整个密码流程必须看起来“受控”，不能像界面卡死

### 6.10 执行中态

要求：

- 操作区成为主反馈区
- 必须显示当前操作类型、进度、当前正在处理的对象
- 必须让用户感知界面仍然活着
- 删除执行中不应写“完成后自动全量重扫”，因为当前行为不是这样

### 6.11 成功反馈态

要求：

- 给出简洁结果
- 告诉用户成功数量 / 失败数量
- 如果有代表性结果，可显示最近一条输出摘要

### 6.12 失败反馈态

要求：

- 失败信息必须可读
- 不要把底层原始错误全部砸到主界面
- 但要保留足够的信息帮助理解

### 6.13 空状态

要求：

- 当没有扫描到任何资产时，要有完整空状态
- 当搜索无结果时，也要有独立空状态提示

## 7. 表现细节规范

### 7.1 焦点与选中态区分

必须区分：

- 当前焦点项
- 已选中项
- 焦点且已选中的项

不能只靠一个标记混过去。

### 7.2 风险标签表达

风险建议的展示建议使用：

- 色彩
- 标签边框 / 胶囊样式感
- 文本权重

但不要过度依赖背景块大面积填充，避免终端中显得脏乱。

### 7.3 大体积与长期未用标记

要求：

- 仍然保留高亮能力
- 但方式更克制
- 不要和风险色冲突
- 可以使用轻量标签、图标、前缀或次级强调

### 7.4 分隔线与边框

要求：

- 统一使用 box-drawing 字符
- 主区域边框可以统一细线
- 危险确认态或核心操作态可以升级成更强边框表现
- 不要把每个小块都做成太重的盒子，避免压迫感

---

## 8. 本地化与排版规则

### 8.1 中英文混排

必须考虑：

- 中文双宽字符
- 英文路径与版本号较长
- Homebrew、npm、uv 等专有名词保持英文

### 8.2 列宽策略

要求：

- 列宽按真实显示宽度计算
- 窄终端下优先保留：名称、大小、最近使用
- 版本和来源可做截断或压缩展示

### 8.3 文案风格

要求：

- 中文界面尽量自然、简洁、非技术化
- 英文界面简洁直接
- 避免说明文字过长撑爆布局

---

## 9. 窄终端退化策略

必须设计窄终端下的退化方案。

例如：

- Header 两行化
- 列表列数减少
- 详情区压缩为更少字段
- Footer 从一行拆成两行
- 操作区文字更短

但仍然不能变成不可读。

---

## 10. 实现建议（基于当前项目）

本次重设计默认继续基于当前 Rust 技术栈实现：

- `ratatui`
- `crossterm`

建议以组件视角重构：

- Header
- List
- Details Panel
- Action Panel
- Footer
- Overlay / Confirm Layer（如果需要）

说明：

- 本次不是换技术栈项目
- 不是重写为 Go / Python / Node
- 重点是基于现有能力做更完整的 TUI 体验落地

---

## 11. 最终交付要求

新的 TUI 重设计最终至少需要输出并落地以下界面：

- Loading
- 默认浏览态
- 搜索态
- 多选态
- 删除确认态
- 清缓存确认态
- 刷新确认态
- 退出确认态
- 权限输入态
- 执行中态
- 成功反馈态
- 失败反馈态
- 空状态

并满足：

- 核心能力不变
- 快捷键语义不变
- 产品行为不变
- 仅操作体验和视觉表达升级
