## 1. 全局要求

在本仓库内执行任何任务前，代理应优先阅读并遵守：

- `docs/AGENT_BEHAVIOR_RULES.md`

如果本文件与其他临时说明发生冲突，优先保证：

- 不误导用户
- 不过早下结论
- 不把“代码层完成”误说成“用户结果已生效”

---

## 2. 必须遵守的核心行为规则

### Rule A：区分三层完成状态

任何会影响用户实际运行结果的修改，都必须明确区分：

1. 源码已修改
2. 测试 / 编译已通过
3. 用户当前运行的版本已经更新，并能看到效果

在没有完成第 3 步前，不得使用：

- “改好了”
- “已经生效了”
- “你现在看到的就是新效果”

### Rule B：TUI / UI 改动必须做可见结果验证

所有 TUI / UI 视觉层改动，都不能只依赖编译通过，必须额外检查：

- 默认态是否可见
- 焦点态是否可见
- 选中态是否可见
- 焦点 + 选中叠加态是否仍然可见
- 颜色是否符合既定语义
- 关键反馈是否会被误认为卡住

### Rule C：测试通过不能替代交付验证

“测试通过”只代表开发验证通过，不代表用户已经拿到正确结果。

以后凡是向用户汇报“验证通过”，必须说明验证层级：

- 代码层验证
- 安装层验证
- 用户可见结果验证

### Rule D：先确认，再下结论

在输出“完成了”“修好了”“已经生效”之前，必须先确认：

- 是否改到了正确文件 / 正确位置
- 是否覆盖到了用户当前正在使用的运行路径
- 是否完成重新安装或更新
- 是否真的符合用户的具体要求

如果其中任一项未完成，必须明确说“还差哪一步”，不能提前下结论。

---

## 3. 对用户的标准汇报格式

当改动会影响用户实际运行结果时，汇报时尽量按以下结构：

- 已修改源码：哪个文件
- 已验证：测试 / 编译是否通过
- 已安装：是否已更新到用户当前运行的版本
- 用户可见变化：现在运行后应该看到什么

---

## 4. 规则追加机制
